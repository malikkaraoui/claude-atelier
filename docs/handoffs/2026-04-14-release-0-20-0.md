### Risques critiques

#### 1. Risque n°1 — écart entre tests synthétiques et runtime réel Claude

Le bug d'origine n'était pas un bug purement algorithmique, c'était un bug de **capture de vérité runtime**. Or tes tests `test/hooks.js` mockent le JSON stdin du hook avec des cas propres :

- `model` live présent,
- transcript contenant littéralement `Set model to ...`,
- cache `/tmp` contrôlé.

Ça valide bien la logique shell, **pas forcément le contrat réel de l'hôte**. Si le vrai `UserPromptSubmit` n'injecte pas `model` comme tu le supposes, ou si le transcript diffère légèrement du format testé, tu peux retomber sur `cache` ou `inconnu` alors que les tests sont verts.

Et tu as déjà un signal inquiétant dans ton contexte : tu mentionnes un prompt système cassé avec `| //')\n66\t` + bannière `MODÈLE INCONNU`. Je ne peux pas affirmer la cause exacte sans reproduction live, mais je peux affirmer ceci : **tant que tu n'as pas fait un aller-retour réel `/model sonnet` → `/model opus` → message normal dans la vraie intégration, le fix n'est pas fermé au niveau produit.**

Ce point est le seul vrai bloqueur release à mes yeux.

#### 2. Risque n°2 — `validate-handoff.js` v5 est plus dur, mais encore asymétrique

Le durcissement est bon :

- `reviewedRange` obligatoire,
- `fromSha` ancêtre de `toSha`,
- `toSha` ancré au premier vrai commit d'intégration.

Ça ferme bien la triche documentaire grossière que tu visais.

En revanche, il reste une asymétrie produit : le validateur sait rejeter un handoff **après intégration**, mais le draft normal est par définition invalide tant qu'il n'a pas été intégré. Ce n'est pas un bug en soi — tu l'assumes explicitement — mais ça veut dire que la mécanique reste fragile si un futur script ou un futur humain commence à traiter `validate-handoff.js` comme un validateur de "draft prêt". Pas bloquant pour 0.20.0, mais le contrat doit rester très clair.

Je ne bloquerais pas la release pour ça.

#### 3. Risque n°3 — saut npm de 15 versions

Le saut de versions n'est pas en soi une surface d'attaque. Ce qui compte, c'est le contenu publié.

Sur ce que je vois dans `package.json`, le package publie :

- `bin/`,
- `src/`,
- `scripts/`,
- `hooks/`,
- `test/`,
- `LICENSE`, `README.md`, `CHANGELOG.md`.

Je ne vois pas de secret, token, ni artefact sensible embarqué dans ce set. `SECURITY.md` n'est pas dans `files`, mais c'est un sujet de doc packaging, pas un sujet sécurité release-critical.

Le vrai angle ici n'est pas la sécurité : c'est la **compatibilité d'upgrade** pour des projets clients déjà installés avec une vieille structure `.claude/`. Comme ton package est un atelier CLI/config plus qu'une lib runtime classique, le risque principal est un projet client partiellement mis à jour, pas un exploit. Là encore : pas un bloqueur 0.20.0 si ton flux recommandé reste `init/update` et non “copie manuelle au petit bonheur”.
